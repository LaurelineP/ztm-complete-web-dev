Git and Git Solutions

Git is a tool that allows developers to keeps tacks of their code
through times and also allows to efficiently collaborate as a team. Thanks to Git any dev in a team is able to add her/his feature despite and to be able to quickly identify if there are conflicts ( old codes vs new codes, or colleagues code working on the same files than yourself). This controls the version of the codebase for each team member

Git Deep Dive

Glossary
  • Git:
    command line providing tools to keep track of any code changes

  • Source Control:
    a provider hosting the codebase

  • Remote:
    remote refers to the hosted code in a source control

  • Repository:
    folder containing the code / the project

  • Branch:
    is like a parallel version of the mutual code on which any changes could be done

  • Merge:
    Fusing code together

  • Commit:
    Save a stable code with a message to keep track

  • Conflict:
    when a code lines are encountering other "new" line from maybe another branch
    This will highlights the conflicting lines in order to decide what to do ( add, remove, keep both code...)

  • Clone a repo:
    Copy locally a repository

  • Fork a repo:
    Copy a repo to your own repository

  • Merge Request:
    Request the repo to add a feature and request for validation - term used for instance with Github

  • Pull Request:
    Request the repo to add a feature and request for validation - term used for instance with Gitlab

  • Fork a repository:
    Copy a repo to your own repository

Commands

Those are the most used command that generally are sufficient to work with.
Please keep in mind that the commands are case sensitive.

  • Git commands start with git <command>
  • Git commands have modifiers:
    modifiers are like various case of the same command
    like commit being the command we can specify -m to add a message, or -a to adjust last commit
  • Commands can be followed by one or multiple modifiers
  • git clone <repo-url>
    copies a repository from the source control ( github, bitbucket, gitlab )

  • git config
    allows to configure your details as a git user

  • git status
    allows to check what are the current changed file

  • git log
    allows to see all prior commits

  • git fetch
    gets all updates from branch

  • git branch
    list all local branches

  • git branch -r or git branch --remote
    list all local branches from the remote repository

  • git branch -d
    delete the specified branch locally

  • git branch -D
    delete the specified branch on the remote repository

  • git checkout -b or git checkout --branch
    From current branch go to the stated one

  • git add
    select following files and/or folders to a staged state in order to apply an action in a second steps upon

  • git stash
    allows to preserve/save on the any changes from the selected files

  • git stash apply
    apply last stashed changes

  • git stash pop
    apply and remove last stashed changes

  • git stash drop
    drop last changes stashed

  • git commit -m or git commit --message
    followed by a string for the messages, save current staged file within that checkpoint. This will highlights the conflicting lines in order to decide what to do ( add, remove, keep both code...)

  • git commit -a or git commit --amend
    Allows to edit last commit

  • git commit -a or git commit --amend
    Allows to edit last commit

  • git push
    Allows to share the locally saved changes to the remote repository

  • git remote -v
    Checks the current remote repository

Open Source and Contributions

Open Source are projects that are public (everyone can see the code) and opened to contributions. Anyone can contribute if they want as long as they follow the project rules.

Starting steps to contribute
  1. Read the Readme
  2. Check the issues's label an try to find one for "good as first ticket"
  3. Fork the project you want to contribute to
  4. Create a branch to add your code
  5. Push the code
  6. Review your code before doing a PR or MR

A Typical developer workflow

Those concept covered above are the one that you most likely end up using every working day as a developer.Here will be described what is expected from a developer in a company. The tasks to do are generally listed on a board, then developers takes the task and finish them.

Fundamental process
  1. Tickets:
    First thing to do is to check the ticket management system KanBan dashboard ( having columns for the process of a ticket ) and tickets that need to be handled: as a developer you'll take a ticket from a `todo` columns and pass it to 'in progress' - this will communicate the ticket status to colleagues
  2. Create a branch:
    When starting a new feature, you need to create a new branch to implement your solution
  3. Update and communicate
    Once done, the ticket should pass through the next column which is most likely `review` and the code should be pushed
  4. Then repeat

Node Basic Setup

In the direction of being a fullstack developer, you'll hear about Node, NPM, Yarn : as they serve this purpose, as they are part of the JavaScript ecosystem.

Tools explanations
  • Back End and Front End
    What is a back-end ? A back end is an environment allowing to deal with servers to host a website for instance, it allows to handle data, registration and more. The Front end is meant to display an interface for the user to interact but without a server hosting the front-end files there would not be any site to go to.

  • Node:
    Node is a JavaScript runtime engine which basically allows to run JavaScript in the Back-End - JavaScript was originally used only for the front-end interactions and the back-end was built with other programming languages (php, python, etc ...) Having Node as back-end enable the "fullstack" developer role and enable the use of only one programming language to know to build things. Node is great to deal with web servers and real time application.
    • Node Installation

    • Update Node
    • Change Node version
    • Update NPM
  • Package Managers:
    NPM (Node Package Manager) or Yarn are both package manager for Node environment. NPM is delivered when installing Node / Yarn need to be installed. NPM is also a CLI ( command line interface ) that allows a developer to access packages that other developers developed ( we can see them as extra tools to work with)
    • NPM
    • Yarn
  • NPM and Yarn:
    Like mentioned, packages are code that developers shared to avoid you from re-writing a similar logic and to use them out of the box by just installing them using the command line when a package.json file is set - for more information do not hesitate to dive into the documentations
    NPM: npm i <package> or npm install <package>
    Yarn: yarn add <package>

  • package.json file
    The package.json file list all dependencies that your project has - dependencies are the packages your project uses. There are 2 kinds of dependencies:

    ❖ regular dependencies : the ones required by your project to run
    ❖dev-dependencies:
       the ones used by the developer and not mandatory for the project to correctly work.

Starting Setup
  1. Init Project:
    npm init -y: will init a package.json file

  2. Read the generated file and adjust if needed:
    • name: project name
    • version: version of your project
    • description: project description
    • main:this is the JS entry point ( = file that will be served first )
    • script:custom script to use by execution npm run <command-name>

  3. Set and use packages - old ways (require and module.exports)
    • npm install -g browserify:
      globally install browserify (to enable the code to understand
      require keyword on the front-end side).

    • browserify script.js > bundle.js:
      will bundle the file script.js to bundle.js.
      Note: this command need to run at the right directory with the corresponding
      file ( script.js being an example here).

    • Use the bundle:
      then replace or add the file to the HTML script src instead of the
      script.js file here by bundle.js
      will bundle the file script.js to bundle.js
      and import using require
      Note: need to export the file using require and file with the code to be imported.
      Note: Each changes applied after the bundle.js has been generated, you need to run the browserify command with the files.
